# selective Thinking 실패 실험 기록

이 문서는 레포에서 제외한 selective Thinking 실패 실험을 따로 설명합니다. 원본 노트북 대신, 아이디어와 일반 로직만 문서로 남깁니다.

## 한 줄 요약

- 시도한 것: 여러 제출의 답이 갈리는 샘플만 Thinking 모델로 다시 풀기
- 남은 것: 어려운 샘플만 골라 다시 보는 아이디어
- 실패로 본 이유: 로그 저장과 최종 answer 파싱이 불안정했고, 공개 문서의 대표 코드로 쓰기 어렵기 때문

## 실험 요약

| 항목 | 내용 |
|---|---|
| 공개 레포 상태 | 원본 노트북 제거, 문서만 유지 |
| 실험 아이디어 | 여러 submission이 서로 다른 답을 낸 샘플만 다시 추론 |
| 사용 모델 | `unsloth/Qwen3-VL-32B-Thinking-unsloth-bnb-4bit` |
| 중간 산출물 | `thinking.txt`, `main-sub-think-2.csv` 등 |
| 이번 정리에서의 위치 | 실패 실험 기록 |

## 실험이 하려던 일

1. `test.csv`와 여러 submission 파일을 같이 읽습니다.
2. 여러 제출이 모두 같은 답을 낸 문제와 갈린 문제를 나눕니다.
3. 갈린 문제만 Thinking 모델로 다시 풉니다.
4. 새 답을 기존 submission에 덮어써 최종 CSV를 만듭니다.

아이디어 자체는 나쁘지 않았습니다. 전체 test를 다시 돌리는 대신 애매한 문제만 골라 재추론하는 방식이기 때문입니다.

## 공개 레포에서 노트북을 뺀 이유

### 1. 파일명과 실제 역할이 어긋남

파일명만 보면 최고점 코드처럼 보일 수 있습니다. 하지만 실제 최종 제출 파이프라인은 `detection_tta_0.94402.ipynb`로 정리되어 있기 때문에, 이 노트북을 대표 성공 코드로 두면 문서 전체 인과관계가 꼬입니다.

### 2. 로그 기반 파싱이 불안정함

이 노트북은 긴 Thinking 출력을 텍스트 파일로 남긴 뒤 다시 파싱하는 흐름이 있습니다. 이런 구조는 아래 같은 문제가 생기기 쉽습니다.
- 로그 append로 중복 엔트리가 생길 수 있음
- `test_id` 단위 분리가 어긋날 수 있음
- 마지막 a/b/c/d 한 글자만 뽑는 규칙이 깨질 수 있음
- 저장된 로그와 최종 dataframe이 어긋날 수 있음

### 3. 공개 포트폴리오 문서에서 오해를 부름

이 노트북을 README에서 최고점 코드로 소개하면, 실제로는 실패한 실험을 성공 사례로 읽게 됩니다. 그래서 성공 서사와 분리했습니다.

## 남겨둘 만한 점

이 노트북에서 완전히 버리지 않은 부분도 있습니다.

- 여러 제출이 갈리는 샘플만 따로 보는 방식
- 전체 추론 대신 어려운 문제에만 추가 계산을 쓰는 발상
- 후처리 아이디어를 코드로 빠르게 검증한 흔적

즉, 원본 노트북 전체를 공개 레포에 둘 필요는 없지만, 어려운 샘플만 다시 보는 아이디어 자체는 실험 메모로 남길 가치가 있습니다.

## 문서로만 남긴 일반 로직

아래는 실패 실험에서도 문제 없이 설명할 수 있는 일반 로직입니다.

1. 여러 제출 결과를 한 표로 모읍니다.
2. 샘플별로 답이 모두 같은지, 갈리는지 나눕니다.
3. 갈리는 샘플만 추가 계산 대상으로 고릅니다.
4. 추가 계산 결과를 기존 제출에 제한적으로 덮어씁니다.

이 방식은 특정 노트북 구현과 무관하게, 후처리 실험 아이디어로만 이해하면 됩니다.

## 이 문서를 읽는 방법

- 최종 제출 파이프라인을 보고 싶다면 `../notebooks/detection_tta_0.94402.ipynb`와 `../README.md`를 먼저 봅니다.
- 모델 교체와 학습 파라미터 조정이 궁금하면 `model-change-experiment.md`를 봅니다.
- 이 문서는 성공 사례가 아니라 실패 기록으로 읽는 것이 맞습니다.
